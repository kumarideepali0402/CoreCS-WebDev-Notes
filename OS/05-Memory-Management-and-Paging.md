## 1. Why Memory Management?

> **Memory Management:** The OS function that tracks which parts of memory are in use, allocates memory to processes when needed, and reclaims it when no longer needed.

### Goals:
- **Isolation** — protect each process's memory from others
- **Efficiency** — maximize memory utilization
- **Transparency** — each process thinks it has the full address space (virtual memory)
- **Relocation** — a program can run at any physical address

### Key Terms:

| Term | Meaning |
|---|---|
| **Logical (Virtual) Address** | Address generated by CPU — what the program sees |
| **Physical Address** | Actual address in RAM |
| **MMU** | Memory Management Unit — hardware that translates logical → physical |
| **Address Space** | Full set of addresses a process can reference |

```
CPU generates logical address
    ↓
MMU translates → physical address
    ↓
RAM accessed at physical address
```

## 2. Contiguous Memory Allocation

Each process occupies a **single contiguous block** of memory.

### Fixed Partition (MFT — Multiprogramming with Fixed number of Tasks)

Memory is divided into **fixed-size partitions** at system boot.

```
Memory Layout:
┌──────────────┐
│  OS (8 MB)   │
├──────────────┤  ← Partition 1 (16 MB)
│   Process A  │
│   (10 MB)    │
│   [6 MB wasted] ← Internal Fragmentation
├──────────────┤  ← Partition 2 (16 MB)
│   Process B  │
│   (16 MB)    │
├──────────────┤  ← Partition 3 (16 MB)
│    EMPTY     │  ← Partition idle (no process fits or no process waiting)
└──────────────┘
```

**Problem:** Internal fragmentation — allocated partition > actual process size → wasted space inside the partition.

### Variable Partition (MVT — Multiprogramming with Variable number of Tasks)

Partitions are created dynamically — each process gets exactly the memory it needs.

```
Memory after loading A (20MB), B (10MB), C (30MB):
┌──────────────┐
│  OS (8 MB)   │
├──────────────┤
│   A (20 MB)  │
├──────────────┤
│   B (10 MB)  │
├──────────────┤
│   C (30 MB)  │
├──────────────┤
│  Free (12MB) │
└──────────────┘

After B exits:
┌──────────────┐
│  OS (8 MB)   │
├──────────────┤
│   A (20 MB)  │
├──────────────┤
│  Free (10MB) │  ← hole where B was
├──────────────┤
│   C (30 MB)  │
├──────────────┤
│  Free (12MB) │
└──────────────┘
→ External Fragmentation: two free holes (10+12 = 22MB free) but a 20MB process can't fit in either individually
```

### Free Space Allocation Strategies:

| Strategy | How it works | Pro | Con |
|---|---|---|---|
| **First Fit** | Allocate **first hole** large enough | Fast | Fragments beginning of memory |
| **Best Fit** | Allocate **smallest hole** that fits | Minimizes wasted space | Slow (scan all); leaves tiny unusable holes |
| **Worst Fit** | Allocate **largest hole** available | Leaves large leftover holes | Slow; degrades large hole availability |
| **Next Fit** | Like First Fit but starts from last allocation point | Faster than First Fit | More uniform fragmentation |

> **GATE Key Point:** **First Fit** is generally fastest. **Best Fit** produces the smallest leftover hole but the most total fragments. Worst Fit is rarely preferred.

### Compaction

Move all processes to one end of memory to consolidate free holes into one large block.

```
Before compaction:           After compaction:
┌──────────┐                 ┌──────────┐
│  OS      │                 │  OS      │
├──────────┤                 ├──────────┤
│  A       │                 │  A       │
├──────────┤                 ├──────────┤
│  FREE    │                 │  B       │  ← moved up
├──────────┤       →         ├──────────┤
│  B       │                 │  C       │  ← moved up
├──────────┤                 ├──────────┤
│  FREE    │                 │  FREE    │  ← all free space consolidated
├──────────┤                 │  (large) │
│  C       │                 │          │
└──────────┘                 └──────────┘
```

**Problem:** Very expensive — requires copying all process data in memory. Processes must be relocated (need relocation support).

## 3. Fragmentation

| Type | Definition | Occurs in | Solution |
|---|---|---|---|
| **Internal Fragmentation** | Allocated memory > requested memory → wasted space **inside** the block | Fixed partitions, Paging | Smaller page/partition sizes |
| **External Fragmentation** | Total free memory is sufficient but **not contiguous** → large process can't load | Variable partitions, Segmentation | Compaction, Paging |

> **Key Insight:** Paging **eliminates external fragmentation** (no contiguous requirement) but introduces **internal fragmentation** (last page may be partially used).

## 4. Paging

> **Paging:** Divide both physical memory and logical address space into fixed-size blocks. Physical memory blocks = **frames**. Logical memory blocks = **pages**. A process's pages can be loaded into any available frames — no contiguity required.

```
Logical Address Space (process):        Physical Memory (RAM):
┌───────┐                               ┌──────────┐
│ Page 0│ ───────────────────────────→  │ Frame 3  │ ← Page 0 loaded here
├───────┤                               ├──────────┤
│ Page 1│ ──────────────────────╮       │ Frame 0  │ (OS)
├───────┤                       ╰────→  ├──────────┤
│ Page 2│ ──────────────╮              │ Frame 7  │ ← Page 1 loaded here
├───────┤               ╰──────────→   ├──────────┤
│ Page 3│                              │ Frame 2  │ ← Page 2 loaded here
└───────┘                              ├──────────┤
                                       │ Frame 5  │ ← Page 3 loaded here
                                       └──────────┘
```

### Page Table

A **per-process data structure** that maps page numbers → frame numbers.

```
Page Table for Process P:
┌──────────┬──────────────┐
│ Page No. │ Frame No.    │
├──────────┼──────────────┤
│    0     │      3       │
│    1     │      7       │
│    2     │      2       │
│    3     │      5       │
└──────────┴──────────────┘
```

### Key Formulas:

```
Page Size = Frame Size = 2^n bytes  (n = offset bits)

Number of Pages  = Logical Address Space Size / Page Size
Number of Frames = Physical Memory Size       / Frame Size

Logical Address  = (Page Number, Offset)
Physical Address = (Frame Number, Offset)
```

### Example:

```
Given:
  Logical Address Space = 32 bytes (5-bit logical address)
  Physical Memory       = 16 bytes (4-bit physical address)
  Page Size             = 4 bytes  → offset = 2 bits

  Number of pages  = 32 / 4 = 8  → page number uses 3 bits
  Number of frames = 16 / 4 = 4  → frame number uses 2 bits

Logical address: 13 = 01101₂ → Page 3 (011₂), Offset 1 (01₂)
Page Table: Page 3 → Frame 2
Physical address: Frame 2, Offset 1 → 1001₂ = 9
```

### Internal Fragmentation in Paging:

```
Process size = 72,766 bytes
Page size    = 4 KB = 4096 bytes

Pages needed = ⌈72766 / 4096⌉ = 18 pages (18 × 4096 = 73,728 bytes allocated)
Internal fragmentation = 73,728 − 72,766 = 962 bytes wasted in last page

Worst case internal fragmentation = page size − 1 bytes
Average internal fragmentation   = page size / 2
```

## 5. Address Translation in Paging

```
Logical Address (generated by CPU):
┌──────────────────┬──────────────┐
│  Page Number (p) │ Offset (d)   │
└──────────────────┴──────────────┘
         │
         ↓ index into Page Table
┌──────────────────┐
│   Page Table     │  (stored in main memory, base address in PTBR)
│   p → f          │
└──────────────────┘
         │
         ↓ f (frame number)
Physical Address:
┌──────────────────┬──────────────┐
│  Frame Number (f)│ Offset (d)   │
└──────────────────┴──────────────┘
```

**Problem:** Every memory access requires **two memory accesses** — one for the page table, one for the actual data. This doubles the memory access time.

**Solution:** TLB (see Section 6).

### Page Table Base Register (PTBR):
- CPU register that stores the **starting address** of the current process's page table
- On context switch, OS saves/restores PTBR

### Page Table Length Register (PTLR):
- Stores the **size** of the page table
- Used to detect invalid page accesses (page number ≥ PTLR → trap)

## 6. Translation Lookaside Buffer (TLB)

> **TLB:** A small, fast **hardware cache** (typically 64–1024 entries) inside the MMU that stores recently used page-number → frame-number mappings.

```
CPU generates logical address → extract page number p
           ↓
    ┌──────────────┐
    │  TLB lookup  │ (fast — associative/parallel search)
    └──────────────┘
         /        \
      HIT          MISS
      (p found)    (p not in TLB)
        |               |
   Get frame f     Access Page Table in memory
   directly        Get frame f → update TLB
        |               |
        └───────────────┘
                 ↓
         Physical Address = (f, d)
         Access actual data in RAM
```

### Effective Memory Access Time (EMAT):

```
Let:
  α = TLB hit ratio (fraction of accesses that hit in TLB)
  t = TLB lookup time
  m = memory access time

EMAT = α × (t + m) + (1 − α) × (t + 2m)
     = t + m + (1 − α) × m
     = t + (2 − α) × m
```

### Example:

```
TLB access time = 20 ns
Memory access time = 100 ns
TLB hit ratio = 80% = 0.8

EMAT = 0.8 × (20 + 100) + 0.2 × (20 + 100 + 100)
     = 0.8 × 120 + 0.2 × 220
     = 96 + 44 = 140 ns

Without TLB: 2 × 100 = 200 ns (page table + data)
Speedup: 200 / 140 ≈ 1.43×
```

> **GATE Key Point:** TLB is an **associative memory** — all entries searched simultaneously in parallel (not sequentially). Also called **associative registers**.

### TLB with ASID (Address Space Identifier):

Each TLB entry can store an ASID — the process ID. This allows TLB entries from **multiple processes** to coexist, avoiding full TLB flush on context switch.

## 7. Multi-Level Paging

For large logical address spaces (32-bit → 2³² bytes = 4 GB), the page table itself becomes huge.

```
32-bit address, 4KB page size:
Number of pages = 4GB / 4KB = 2^20 = 1,048,576 pages
Page table size = 1M entries × 4 bytes = 4 MB PER PROCESS → too large to keep in RAM contiguously
```

**Solution:** Break the page table itself into pages using a multi-level hierarchy.

### Two-Level Paging (32-bit):

```
Logical Address (32-bit, 4KB pages):
┌──────────┬──────────┬──────────────┐
│  p1 (10) │  p2 (10) │  offset (12) │
└──────────┴──────────┴──────────────┘
      │           │
      │           └── index into 2nd-level page table
      └── index into outer (1st-level) page table
```

```
Address Translation:
  PTBR → Outer Page Table[p1] → 2nd-level Page Table[p2] → Frame f
  Physical address = (f, offset)
  Requires 3 memory accesses (outer table + inner table + data)
```

### Memory Savings:

```
Without 2-level paging: 4 MB page table always in memory
With 2-level paging: only the outer table (4KB) + pages of the inner table that are actually used must be in memory
→ For sparse address spaces (most real programs), most inner page tables are never created → huge savings
```

### For 64-bit Systems:

- Two levels are not enough (page table still too large)
- Modern OSes use **4-level paging** (x86-64: PGD → PUD → PMD → PTE)

## 8. Segmentation

> **Segmentation:** Divide the logical address space into **variable-size** segments corresponding to logical units: code segment, data segment, stack segment, heap segment, etc.

```
Segment Table:
┌──────────┬───────┬────────┐
│ Segment  │ Base  │ Limit  │
├──────────┼───────┼────────┤
│    0     │ 1400  │  1000  │  ← code segment starts at 1400, length 1000
│    1     │ 6300  │  400   │  ← data segment
│    2     │ 4300  │  400   │  ← stack segment
└──────────┴───────┴────────┘

Logical Address: (segment s, offset d)
If d < Limit[s]:
    Physical Address = Base[s] + d
Else:
    Trap (segmentation fault)
```

### Segmentation vs Paging:

| Feature | Paging | Segmentation |
|---|---|---|
| Division | Fixed-size pages | Variable-size segments |
| Logical structure | No (arbitrary split) | Yes (code, data, stack) |
| External fragmentation | No | Yes |
| Internal fragmentation | Yes (last page) | No |
| Address | (page, offset) | (segment, offset) |
| Sharing | Difficult | Easy (share a segment) |
| Protection | Per-page bits | Per-segment (RWX) |

> **Real systems (e.g., x86 in protected mode)** use **segmentation + paging** combined: segment → linear address → paging → physical address.

## 9. Virtual Memory & Demand Paging

> **Virtual Memory:** The OS gives each process the illusion of having a large, private address space — even if physical RAM is smaller. Pages are loaded into RAM **on demand**.

### Demand Paging

Pages are only loaded into memory **when they are accessed** (not all at program start).

```
Page access:
  CPU references logical address
      ↓
  MMU checks page table — is valid bit set?
      ↓ NO (page not in memory)
  PAGE FAULT trap → OS takes control
      ↓
  OS finds a free frame (or evicts a page)
      ↓
  OS reads the required page from disk (swap space)
      ↓
  Update page table (set valid bit, frame number)
      ↓
  Resume the faulting instruction
```

### Page Fault Handling Steps:

```
1. CPU accesses a page → invalid bit in page table → PAGE FAULT interrupt
2. OS saves process state
3. OS determines if access is valid (legal address?) or illegal (segfault?)
4. Find a free frame in memory (or choose a victim page to evict)
5. Schedule disk I/O to read the page from swap into the frame
6. When I/O completes: update page table entry (frame #, valid bit = 1)
7. Restart the faulting instruction
```

### Effective Access Time with Page Faults:

```
Let:
  p = page fault rate (0 ≤ p ≤ 1)
  m = memory access time (e.g., 200 ns)
  page_fault_time = time to handle a page fault (e.g., 8 ms = 8,000,000 ns)

EAT = (1 − p) × m + p × page_fault_time
    = (1 − p) × 200 + p × 8,000,000

For p = 0.001 (1 in 1000 accesses faults):
EAT = 0.999 × 200 + 0.001 × 8,000,000
    = 199.8 + 8000 = 8199.8 ns ≈ 8.2 μs
    (40× slowdown compared to no page faults)

→ Keep page fault rate VERY LOW for acceptable performance
```

## 10. Page Replacement Algorithms

When memory is full and a new page must be loaded, the OS must **evict (replace)** an existing page. Goal: minimize page faults.

### Reference String Setup:

```
Reference string: 7 0 1 2 0 3 0 4 2 3 0 3 2 1 2 0 1 7 0 1
Number of frames: 3
```

### Algorithm 1 — FIFO (First-In, First-Out)

Evict the page that has been in memory **the longest**.

```
Reference: 7 0 1 2 0 3 0 4 2 3 0 3 2 1 2 0 1 7 0 1
Frames:    3

Step  Ref  Frames        Fault?
  1    7   [7, -, -]     YES (cold miss)
  2    0   [7, 0, -]     YES
  3    1   [7, 0, 1]     YES
  4    2   [2, 0, 1]     YES  evict 7 (oldest)
  5    0   [2, 0, 1]     NO
  6    3   [2, 3, 1]     YES  evict 0 (oldest)
  7    0   [2, 3, 0]     YES  evict 1 (oldest)
  8    4   [4, 3, 0]     YES  evict 2 (oldest)
  ...

Total page faults: 15
```

**Belady's Anomaly:** With FIFO, **more frames can cause MORE page faults** (counterintuitive). FIFO does not have the stack property.

```
Example: Reference: 1 2 3 4 1 2 5 1 2 3 4 5
  3 frames → 9 page faults
  4 frames → 10 page faults  ← MORE faults with MORE frames!
```

### Algorithm 2 — Optimal (OPT / MIN)

Evict the page that will **not be used for the longest time in the future**.

```
Reference: 7 0 1 2 0 3 0 4 2 3 0 3 2 1 2 0 1 7 0 1
Frames: 3

At step 4 (page 2, frames = [7,0,1]):
  Next use of 7: step 18
  Next use of 0: step 5
  Next use of 1: step 14
  Evict 7 (used furthest in future)
  Frames: [2, 0, 1]

Total page faults: 9  ← minimum possible
```

**Problem:** Requires knowing the **future** — not implementable in practice. Used as a **theoretical benchmark** to evaluate other algorithms.

### Algorithm 3 — LRU (Least Recently Used)

Evict the page that was **least recently used** (used furthest in the past).

```
Reference: 7 0 1 2 0 3 0 4 2 3 0 3 2 1 2 0 1 7 0 1
Frames: 3

Step  Ref  Frames (most→least recent)    Fault?
  1    7   [7]                            YES
  2    0   [0, 7]                         YES
  3    1   [1, 0, 7]                      YES
  4    2   [2, 1, 0]     evict 7 (LRU)   YES
  5    0   [0, 2, 1]     (0 was in memory, promote) NO
  6    3   [3, 0, 2]     evict 1 (LRU)   YES
  ...

Total page faults: 12
```

**LRU does NOT suffer from Belady's Anomaly** — it has the stack property (n+1 frames ⊇ pages in n frames).

### LRU Implementation:

| Method | How | Cost |
|---|---|---|
| **Counter-based** | Timestamp each access; evict min-timestamp | O(n) search on fault |
| **Stack (doubly linked list)** | Move accessed page to top; evict from bottom | O(1) eviction, O(1) update |
| **Reference bits (approx.)** | 1-bit per page; periodically shift right; evict lowest | Hardware support needed |

### Algorithm 4 — LFU (Least Frequently Used)

Evict the page with the **lowest access count**.

```
Problem: A page heavily used early (high count) but no longer needed stays in memory
         → Old pages with high counts never get evicted
Fix: Aging — periodically shift counter right (halve counts)
```

### Algorithm 5 — Clock (Second Chance)

Approximation of LRU using a **reference bit** and a circular "clock hand".

```
Each page in memory has a reference bit (R):
  R = 1: page was recently accessed
  R = 0: page has not been recently accessed

On page fault:
  While true:
    If R[hand] == 0:
      Evict this page → replace with new page, advance hand
    Else:
      R[hand] = 0  (give it a second chance)
      Advance hand

Pages recently accessed get one more chance before eviction.
```

### Comparison:

| Algorithm | Page Faults (3 frames, above example) | Notes |
|---|---|---|
| **FIFO** | ~15 | Simple; Belady's anomaly |
| **Optimal** | ~9 | Best possible; impractical |
| **LRU** | ~12 | Good; no Belady's anomaly; hardware-costly |
| **Clock** | ~13–14 | LRU approximation; practical |
| **LFU** | Varies | Suffers from frequency bias |

> **GATE Key Point:** OPT ≤ LRU ≤ Clock ≤ FIFO (in terms of page faults, approximately). OPT is the lower bound.

## 11. Thrashing

> **Thrashing:** A process spends more time **paging (swapping pages in/out)** than executing — CPU utilization collapses.

### Cause:

```
Too many processes in memory, each with too few frames allocated
→ Each process constantly page-faults
→ All processes waiting on disk I/O
→ CPU appears idle → OS loads MORE processes (making it worse)
→ Thrashing death spiral
```

```
         ┌─────────────────────────────────────┐
         │         CPU Utilization              │
         │                 ███                  │
         │               ██   ██                │
         │             ██       ██              │
CPU util │           ██           ██            │
         │         ██               ██          │
         │       ██                   ██        │
         └──────────────────────────────────────┘
                              ↑
               Degree of Multiprogramming
           (optimal: peak of curve — beyond this → THRASHING)
```

### Solutions:

| Solution | How |
|---|---|
| **Working Set Model** | Track pages used in last Δ time units (working set). Allocate enough frames to hold the working set. If sum of all working sets > total frames → suspend a process. |
| **Page Fault Frequency (PFF)** | If page fault rate too HIGH → give process more frames. If too LOW → reclaim frames. Keep all processes' fault rates in acceptable range. |
| **Reduce multiprogramming** | Suspend/swap out some processes to give remaining processes more frames |
| **Increase physical memory** | Most effective long-term solution |

### Working Set Model:

```
Working Set W(t, Δ) = set of pages referenced in the time interval (t-Δ, t)

Δ = working set window (tunable parameter)
  Too small → doesn't capture locality
  Too large → encompasses multiple localities
  Infinite → entire program

Total frames needed = Σ |W(t, Δ)| for all processes
If > total frames → suspend a process
```

## 12. FAANG / MAANG Interview Questions

> Questions asked at Google, Amazon, Meta, Microsoft, Apple, and other top companies.

---

### Memory Management Fundamentals

**Q1. What is the difference between logical and physical address? What translates between them?**
> A **logical (virtual) address** is what the CPU generates — the address the running program sees. A **physical address** is the actual location in RAM. The **MMU (Memory Management Unit)** — a hardware component — translates logical → physical using the page table. This abstraction allows each process to think it has a private address space, enables memory protection (a process can't access another's physical memory), and enables virtual memory (logical address space can be larger than physical RAM).

**Q2. What is internal vs external fragmentation? Which does paging solve?**
> **Internal fragmentation** = allocated memory > needed memory — wasted space *inside* an allocation (e.g., last page in paging is only half full). **External fragmentation** = total free memory is sufficient but scattered in non-contiguous holes so a large allocation can't fit. **Paging eliminates external fragmentation** (any free frame works, no contiguity needed) but introduces **internal fragmentation** (last page partially used, up to page_size − 1 bytes wasted). Segmentation eliminates internal fragmentation but suffers external fragmentation.

**Q3. Why do we need multi-level page tables? How many levels does x86-64 use?**
> For a 32-bit address space with 4KB pages, the page table has 1 million entries × 4 bytes = **4 MB per process**, all of which must be contiguous in memory. Most processes use only a small fraction of their address space, so most entries are invalid. Multi-level paging solves this by making the page table itself pageable — only the parts of the page table that map used address ranges need to be in RAM. **x86-64 uses 4 levels** (PGD → PUD → PMD → PTE), handling 48-bit virtual addresses (the other 16 bits are sign-extended). Some kernels now use 5-level paging for 57-bit addresses (Linux since 5.5).

---

### TLB and Performance

**Q4. What is a TLB miss? What happens when a TLB miss occurs?**
> A **TLB miss** happens when the MMU looks up a page number in the TLB (associative cache) and doesn't find it. On a miss: (1) the MMU does a **hardware page table walk** — reads the multi-level page table from memory using the PTBR; (2) loads the found page-table entry into the TLB (evicting an old entry if full); (3) retries the memory access. On **software-managed TLBs** (MIPS architecture), the CPU raises a TLB miss exception and the OS walks the page table and reloads the TLB. TLB misses are far cheaper than page faults (no disk I/O) — just extra memory accesses — but still add latency.

**Q5. What is TLB shootdown? When does it happen?**
> When a page table entry is modified (e.g., a page is unmapped or permission changed), the corresponding TLB entry in **every CPU core** must be invalidated — otherwise stale entries cause incorrect translations or security violations. This process is called a **TLB shootdown**: the modifying CPU sends an inter-processor interrupt (IPI) to all other cores, which then each execute a `INVLPG` (or flush their entire TLB), then acknowledge. TLB shootdowns are expensive in multi-core systems — a single `munmap()` call can cause all CPU cores to stall. This is why processes avoid frequent large unmaps.

---

### Page Replacement

**Q6. Explain LRU page replacement. What are the practical implementation challenges?**
> LRU (Least Recently Used) evicts the page accessed least recently. It's based on the **locality principle** — recently used pages are likely to be used again. **Implementation challenges:** (1) **Counter-based:** add a 64-bit timestamp per page-table entry, updated on every memory access — requires hardware support on every single memory reference (prohibitive overhead). (2) **Stack-based:** maintain a doubly-linked list — expensive to update on every access. In practice, **approximate LRU** is used: Linux uses a **clock algorithm** variant with reference bits (set by hardware on access, cleared periodically by OS) — fast and cheap, good approximation.

**Q7. What is Belady's Anomaly? Which algorithm suffers from it?**
> **Belady's Anomaly** is the counterintuitive phenomenon where **adding more frames causes MORE page faults**. It is specific to the **FIFO** page replacement algorithm. This happens because FIFO evicts based on age, not usefulness — more frames can change which pages survive, sometimes in a worse order. Algorithms with the **stack property** (LRU, OPT) do NOT exhibit Belady's Anomaly — adding a frame can only reduce or maintain the same page fault count, never increase it. The stack property means: the set of pages in memory with n frames is always a subset of the set with n+1 frames.

---

### Virtual Memory and Demand Paging

**Q8. Walk me through exactly what happens when a page fault occurs.**
> (1) CPU references a virtual address — the MMU finds the page-table entry with the **valid bit = 0** (page not in RAM). (2) MMU raises a **page fault exception** — CPU switches to kernel mode and jumps to the page fault handler. (3) OS checks if the address is valid (within the process's virtual address space). If not → `SIGSEGV` (segmentation fault). (4) OS selects a free frame. If no free frame: run page replacement algorithm to select a victim page; if victim is dirty (modified), write it to swap first. (5) OS reads the required page from disk (swap or executable file) into the frame — process is blocked during I/O. (6) OS updates the page table (set frame number, valid bit = 1). (7) Resume the faulting instruction from the beginning.

**Q9. What is copy-on-write (COW)? How does it relate to fork()?**
> **Copy-on-Write (COW)** is an optimization where, after `fork()`, the parent and child **share the same physical pages** — both their page tables point to the same frames, but all pages are marked **read-only**. No actual copying occurs at fork time. When either process tries to **write** to a shared page, the MMU detects the write to a read-only page, triggers a fault, and the OS then creates a private copy of that page for the writing process. This makes `fork()` very fast (O(1) regardless of address space size) for processes that `exec()` immediately (like shells), since most pages are never written and therefore never copied.

### System Design / Scenario Questions

**Q10. How does the Linux kernel manage memory for user processes? (high level)**
> Linux uses a **demand-paged virtual memory** system. Each process has a virtual address space divided into regions tracked by **VMAs (Virtual Memory Areas)** — `vm_area_struct` objects linked in a red-black tree for fast lookup. Physical pages are tracked by `struct page` in the **page cache**, shared between file-backed pages and anonymous pages. Page replacement uses a **two-list LRU** (active + inactive lists), with pages migrating between lists based on reference bits. The kernel uses **slab allocator** for kernel objects and **buddy allocator** for page-granule allocation. On memory pressure, the **kswapd** daemon runs page reclaim in the background, and **OOM killer** terminates processes if physical memory is truly exhausted.

**Q11. A process accesses memory in a tight loop but performance is terrible. Diagnosis?**
> Likely causes in order of probability: (1) **Cache thrashing** — working set doesn't fit in L1/L2/L3 cache, causing constant cache misses. Check with `perf stat -e cache-misses`. (2) **TLB thrashing** — many distinct pages accessed, exceeding TLB capacity. Use huge pages (`madvise(MADV_HUGEPAGE)`) to reduce TLB pressure. (3) **Page thrashing** — not enough RAM, process is constantly paging to disk. Check `vmstat` for high `si/so` (swap in/out). (4) **NUMA effects** — process on CPU 0 accessing memory local to CPU 1 → remote memory latency. Check `numastat`. Fix by pinning process to a NUMA node (`numactl`). (5) **False sharing** — multiple CPU cores' cache lines overlap a hot variable in a struct; add padding.

**Q12. What is the difference between swap space and a memory-mapped file?**
> **Swap space** is disk space used as an overflow for **anonymous pages** (heap, stack, BSS) — pages that have no file backing. When a dirty anonymous page is evicted, it's written to swap; when needed again, read back. **Memory-mapped files** (`mmap()`) map a file directly into the virtual address space. Accesses read/write the file via the page cache — the file itself is the backing store, so the page can be evicted by simply discarding it if clean (or writing it back if dirty). The key difference: anonymous pages lose data if not written to swap; file-backed pages are durable — evicting them doesn't lose data (the file still has the content). `mmap` avoids a `read()`→user buffer copy by mapping the kernel's page cache directly, reducing copies for large file I/O.

### Quick Reference

**Address Translation Formula:**
```
Logical Address  = Page Number × Page Size + Offset
Physical Address = Frame Number × Frame Size + Offset
Need  = Max − Allocation

Page size = 2^d where d = number of offset bits
```

**EMAT with TLB:**
```
EMAT = α(t + m) + (1-α)(t + 2m)     [single level page table]
     = t + m + (1-α)m
where α = hit ratio, t = TLB time, m = memory time
```

**EMAT with Page Faults:**
```
EAT = (1-p) × m + p × page_fault_service_time
where p = page fault rate
```

**Fragmentation Summary:**
```
Fixed partitions     → Internal fragmentation
Variable partitions  → External fragmentation
Paging               → Internal fragmentation (last page), eliminates external
Segmentation         → External fragmentation, eliminates internal
```

**Page Replacement Algorithm Ranking (fewest page faults):**
```
OPT (theoretical minimum) ≤ LRU ≤ Clock ≤ FIFO
```

**Belady's Anomaly:**
```
Only FIFO suffers from it.
LRU and OPT have the STACK PROPERTY → more frames = fewer or equal faults.
```

**Thrashing Fix Summary:**
```
Working Set Model     — allocate frames ≥ working set size per process
Page Fault Frequency  — adjust frames based on measured fault rate
Reduce multiprogramming — suspend processes to free frames
```
